# SpotLine 콘텐츠 기반 플랫폼 전환 전략 제안서

## 배경

### 현재 문제
- "SpotLine 시작하기" 버튼은 백엔드 API → 파트너 매장 리다이렉트 구조
- 매장 영업/온보딩이 선행되어야만 서비스가 작동
- 물리적으로 영업에 시간이 많이 소요됨

### 전환 방향
**영업 기반 → 콘텐츠 기반 플랫폼**
- 크루원들이 admin에서 블로그처럼 장소 콘텐츠를 직접 작성/업로드
- SEO를 통한 유기적 트래픽 확보로 인지도 구축 (2026년 목표)
- 인지도 확보 후 매장을 파트너로 자연스럽게 전환

## 현재 코드베이스 분석

### 기존 플로우
```
"시작하기" 클릭 → GET /api/experience → window.location.href = redirectUrl → /spotline/[storeId]
```
- 백엔드에 등록된 매장이 있어야만 동작
- 콘텐츠 기반 전략과 맞지 않음

### 기존 데이터 모델에서 활용 가능한 부분
- `spotlineStory` 필드: `{ title, content, tags[] }` 구조가 이미 존재
- `NextSpot`: 다음 장소 추천 카드 구조 존재
- `StoreCategory`: cafe, restaurant, exhibition, hotel, retail, culture, other
- `representativeImage`: 대표 이미지 필드 존재

### 현재 없는 것
- Admin 시스템 (라우트, 인증, 대시보드 없음)
- 블로그/콘텐츠 관련 페이지 없음
- 작성자(author) 메타데이터 없음
- 발행일/버전 관리 없음
- SEO용 slug 체계 없음
- 리치 텍스트/마크다운 본문 없음
- 다중 이미지 지원 (legacy Store에는 images[] 있으나 현재 SpotlineStore에는 단일 이미지)

### 주요 파일
| 파일 | 역할 |
|---|---|
| `src/app/page.tsx` | 랜딩 페이지 (Hero + CTA + How-to) |
| `src/components/spotline/SpotlineExperienceButton.tsx` | "시작하기" 버튼 (현재 API 호출 방식) |
| `src/app/spotline/[qrId]/page.tsx` | 메인 Spot 페이지 |
| `src/types/index.ts` | 전체 타입 정의 |
| `src/lib/api.ts` | API 레이어 |
| `src/app/api/demo/store/route.ts` | 데모 데이터 |

## 제안: "시작하기" 플로우 3가지 방향

### 방향 A: 지역 기반 탐색 피드
```
"시작하기" → 위치 허용 → /explore (근처 Spot 피드)
```
- 크루가 올린 장소들이 카드형으로 나열
- 카테고리/거리 필터
- 각 카드 클릭 시 상세 콘텐츠 페이지 (`/spots/[slug]`)
- **장점**: 직관적, 위치 기반 서비스의 자연스러운 UX
- **단점**: 초기 콘텐츠가 적으면 빈 피드 문제, SEO 효과 제한적

### 방향 B: 지역/테마 선택 → 큐레이션
```
"시작하기" → 지역 또는 테마 선택 → /spots/seongsu, /theme/cafe-tour 등
```
- 먼저 지역(성수, 연남, 을지로 등) 또는 테마(카페투어, 전시, 맛집)를 선택
- 해당 큐레이션 리스트로 이동
- **장점**: SEO에 가장 유리 (각 지역/테마 페이지가 검색에 잡힘), 콘텐츠 구조화 용이
- **단점**: 초기에 지역별 콘텐츠 최소 수량 필요

### 방향 C: 기존 방식 유지 + 콘텐츠 병행
```
"시작하기" → 기존 QR 기반 플로우 유지
+ 별도 "둘러보기" → /spots 또는 /blog
```
- 시작하기는 기존 QR/매장 플로우 그대로
- 랜딩에 "둘러보기" 버튼 추가하여 크루 콘텐츠 피드로 연결
- **장점**: 기존 기능 유지하며 점진적 전환, 리스크 낮음
- **단점**: 두 가지 경로 혼재로 사용자 혼란 가능, "시작하기"가 콘텐츠 없으면 여전히 빈 경험

## 제안: 콘텐츠 페이지 구조 3가지 방향

### 구조 1: 장소 소개 + 다음 Spot 연결
- 현재 `spotlineStory` 구조를 확장
- 장소 소개 글 + 사진 + 근처 다음 Spot 추천
- SpotLine 고유 가치("다음 경험 제안")를 콘텐츠에서도 유지
- 기존 `/spotline/[qrId]` 페이지와 일관된 경험

### 구조 2: 블로그형 리뷰/가이드
- 크루 이름, 작성일, 마크다운 본문, 여러 사진
- 일반 블로그에 가까운 형태
- SEO에 유리하지만 SpotLine만의 차별화 약함

### 구조 3: 루트(경로) 중심 콘텐츠
- 여러 Spot을 묶어 하나의 루트로 구성 (예: '성수 카페 3곳 루트')
- SpotLine의 "다음 장소" 컨셉과 가장 잘 맞음
- 사용자가 루트를 따라가며 여러 장소를 탐색
- SEO에도 유리 (롱테일 키워드: "성수 카페 추천 코스")

## 제안: Admin 시스템 위치

### 옵션 1: 이 프로젝트에 /admin 라우트로 포함
- 같은 Next.js 프로젝트 내 `/admin` 경로에 크루용 콘텐츠 작성 페이지
- 빠르게 시작 가능, 타입/컴포넌트 공유
- 프로젝트가 커질 수 있음

### 옵션 2: 별도 프로젝트/레포로 분리
- Admin은 별도 프론트엔드로 분리
- 이 프로젝트는 사용자 대면 콘텐츠 표시에만 집중
- 초기 셋업 비용 높지만 장기적으로 깔끔

### 옵션 3: 나중에 결정
- 우선 사용자 대면 콘텐츠 페이지 구조부터 잡고
- Admin은 이후에 결정

## 결정 필요 사항 (다음 논의 시)

1. **"시작하기" 플로우**: 방향 A / B / C 중 선택
2. **콘텐츠 페이지 구조**: 구조 1 / 2 / 3 중 선택 (혼합도 가능)
3. **Admin 시스템 위치**: 옵션 1 / 2 / 3 중 선택
4. **우선 구현 범위**: 랜딩 페이지 변경부터? 콘텐츠 페이지부터? Admin부터?
5. **백엔드 API**: 콘텐츠 CRUD API는 기존 백엔드(localhost:4000)에 추가하는지, 별도 서비스인지

## 개인 의견 (참고)

**방향 B(지역/테마 선택) + 구조 3(루트 중심)**의 조합을 추천합니다.

이유:
- SEO 관점: `/spots/seongsu`, `/routes/seongsu-cafe-tour` 같은 URL이 검색에 잘 잡힘
- SpotLine 정체성: "다음 장소 제안"이라는 핵심 가치를 콘텐츠에서도 살릴 수 있음
- 크루 운영: 루트 단위로 콘텐츠를 만들면 한 번에 여러 장소를 소개할 수 있어 효율적
- 사용자 경험: 단일 장소 소개보다 "코스/루트"가 실질적 가치가 더 높음
- 매장 전환: 루트에 포함된 매장에게 "이미 SpotLine에서 소개되고 있다"는 영업 근거 제공
